# 1. Epic Name

PostgreSQL Persistence Migration and Solution Hygiene

# 2. Goal

## Problem
Tenax currently relies on in-memory persistence for backend behavior, which is sufficient for early prototyping but not reliable for realistic multi-request workflows, data durability, or production-grade validation. In-memory storage resets state on restart and hides persistence-specific concerns such as schema evolution, relational constraints, and query behavior against a real database engine. This creates delivery risk because backend and API behavior may pass local checks while failing once durable storage is introduced. The current solution also contains autogenerated placeholder artifacts (for example, default template files) that increase noise and reduce maintainability.

## Solution
Define and deliver a migration epic that replaces in-memory persistence with PostgreSQL-backed persistence in the backend, implemented through EF Core and managed schema evolution via dotnet ef migrations. The epic will establish the baseline DbContext model, migration workflow, configuration conventions, and environment wiring needed for durable persistence. It will also include solution hygiene tasks to remove autogenerated placeholders and align project structure with intended Clean Architecture responsibilities. This creates a stable persistence foundation for current and future feature epics.

## Impact
Expected outcomes include durable data across runs, improved confidence in backend correctness, fewer persistence-related regressions later in delivery, and cleaner solution structure that is easier to maintain. It should reduce rework in downstream feature implementation and improve test fidelity by validating behavior against PostgreSQL semantics.

# 3. User Personas

- Backend Developer: Needs a reliable persistence layer to implement and verify application behavior using real relational storage.
- Full-Stack Developer: Needs consistent API behavior across local and test environments with durable data.
- QA Engineer: Needs repeatable backend verification against PostgreSQL instead of ephemeral in-memory state.
- Tech Lead/Architect: Needs architecture conformance (Web -> Application -> Domain, Infrastructure implementations) and maintainable project hygiene.

# 4. High-Level User Journeys

- Local Development Journey:
  A developer starts the backend configured for PostgreSQL, runs migrations using dotnet ef, and validates CRUD flows with durable data.
- Schema Evolution Journey:
  A developer introduces a domain/application change requiring persistence updates, creates a new migration, and applies it through the agreed workflow.
- Test Reliability Journey:
  QA/developers execute backend and integration checks against PostgreSQL-backed persistence to catch relational and schema issues early.
- Solution Hygiene Journey:
  The team removes autogenerated placeholder files and keeps the solution aligned to feature-focused, production-intent structure.

# 5. Business Requirements

## Functional Requirements

- The backend must persist data in PostgreSQL rather than in-memory storage for supported features.
- The Infrastructure layer must host EF Core persistence implementations without leaking EF Core dependencies into Domain.
- A primary EF Core DbContext must be defined and wired through dependency injection in the composition root.
- Connection string and provider configuration must support local development and test usage patterns.
- Baseline EF Core migration files must be generated using dotnet ef CLI and committed to source control.
- A documented migration workflow must exist for creating and applying future migrations (including command examples and expected paths).
- Existing supported feature flows (notably flashcard authoring/management) must work with PostgreSQL-backed persistence.
- Persistence abstractions in Application must remain implementation-agnostic while Infrastructure provides concrete repository/database behavior.
- The epic must remove autogenerated placeholder artifacts (for example, template Class1 files) when they do not provide product value.
- The solution/project structure must remain synchronized and clean after cleanup tasks.
- Developer-facing docs must describe prerequisites, startup assumptions, and common troubleshooting for migration/application of schema changes.

## Non-Functional Requirements

- Data Durability:
  - Data must survive application restarts in normal local/backend runtime configurations.
- Performance:
  - Core CRUD operations should remain within acceptable baseline latency for local and test workloads.
- Reliability:
  - Migration application must be deterministic and repeatable on clean environments.
  - Startup failure modes (invalid connection string, unavailable DB, migration mismatch) must be diagnosable.
- Security:
  - Database credentials/secrets must not be hardcoded in source code or committed in plaintext.
  - Development defaults must be clearly separated from production secret management.
- Maintainability:
  - Persistence code must follow Clean Architecture boundaries and avoid cross-layer coupling.
  - Removal of scaffolding noise must improve readability and reduce confusion for new contributors.
- Observability:
  - Backend logs should provide actionable diagnostics for connection and migration failures.

# 6. Success Metrics

- Durability KPI: 100% of designated persistence-backed feature data remains available across backend restarts in local verification scenarios.
- Migration KPI: Baseline database can be created from zero state using documented dotnet ef workflow with no manual SQL edits.
- Quality KPI: Persistence-related defects caused by in-memory vs relational mismatch decrease sprint-over-sprint.
- Maintainability KPI: Autogenerated placeholder files in production code paths are reduced to zero where not intentionally required.
- Team Adoption KPI: Engineers use documented PostgreSQL + migration workflow as the default backend development path.

# 7. Out of Scope

- Implementing new end-user product capabilities beyond persistence migration and cleanup needs.
- Production database hosting architecture, disaster recovery strategy, and full-scale operations hardening.
- Multi-database provider support (PostgreSQL is the only target in this epic).
- Full data backfill/migration from hypothetical legacy production systems.
- Broad refactoring unrelated to persistence or cleanup of obvious autogenerated placeholders.

# 8. Business Value

High.

This epic replaces a prototype persistence approach with a durable relational foundation, reducing downstream risk for all backend features and improving confidence in integration behavior. It also improves engineering efficiency through standardized migration workflows and cleaner solution structure, lowering maintenance overhead and onboarding friction.

# 9. Implementation Status (2026-03-15)

- PostgreSQL persistence implementation is in place with EF Core in Infrastructure and startup migration application in Web.
- Baseline migration artifacts are committed under `src/Tenax.Infrastructure/Persistence/Migrations`.
- Flashcard API contracts remain shape-stable while introducing persistence-specific error semantics (`409 concurrency_conflict`, `503 persistence_unavailable`) for relevant operations.
- Frontend data-layer behavior aligns to contract notes for retry/refetch/rollback behavior under persistence conflict and outage states.
- Solution hygiene cleanup for production code paths is complete for template placeholder artifacts.

# 10. Test Strategy and Prerequisites (Current)

- Unit tests continue to cover domain/application behavior.
- Integration tests use PostgreSQL containers via `Testcontainers.PostgreSql` to validate persistence-backed behavior.
- Backend web integration setup uses custom `WebApplicationFactory<Program>` and starts PostgreSQL before host migrations.
- Prerequisites:
  - .NET 10 SDK.
  - Docker runtime available for integration tests.
  - Valid `ConnectionStrings:Tenax` configuration for runtime/migration execution.
